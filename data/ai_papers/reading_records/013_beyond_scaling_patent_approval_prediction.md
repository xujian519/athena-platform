### 📊 论文概览

**研究类型**: 方法论论文 + 专利授权预测 + LLM局限性分析

**核心贡献**:
1. **FLAN Graph** - 细粒度权利要求依赖图,捕获权利要求间的依赖关系
2. **LLM扩展性研究** - 全面测试多种LLM在专利授权预测上的表现
3. **发现LLM局限性** - 证明简单扩展模型规模并不保证性能提升
4. **SOTA超越** - GraphSage + FLAN Graph超越原有SOTA 7.4% AUC

### 🎯 核心内容

#### 3.1 研究动机

**核心问题**: 模型扩展是否总是有效？

```
专利授权预测任务:
├─ 目标: 预测每个权利要求是否会被USPTO授权
├─ 挑战:
│   ├─ 时间敏感性: 新颖性评估取决于申请日期
│   ├─ 结构依赖性: 权利要求之间存在依赖关系
│   ├─ 知识密集性: 需要技术和法律知识
│   └─ 结果不一致: 审查员主观判断带来偏差
└─ 意义: 知识产权保护占美国GDP 40%, 就业30%
```

**SOTA基线**: BERT嵌入 + 手工特征 (Gao et al., ACL 2022)

#### 3.2 数据集: PATENTAP

**数据来源**:
- USPTO公开数据
- 按申请日期排序,历史数据训练,新数据评估

**数据规模**:
| 指标 | 训练集 | 验证集 | 测试集 |
|------|--------|--------|--------|
| 权利要求数 | 1,485,693 | 278,215 | 185,477 |
| 申请数 | 87,883 | 16,955 | 11,148 |
| 授权率 | 81.36% | 83.41% | 84.92% |

**数据特点**:
- 平均每个申请17个权利要求
- 92%的权利要求<128 tokens
- 平均长度54 tokens
- 高度不平衡 (大部分授权)

#### 3.3 LLM扩展性实验

**测试模型**:
```
开放源代码LLM:
├─ LLaMA2 (7B, 13B, 70B)
├─ Mistral (7B)
└─ Vicuna (7B, 13B, 70B)

闭源LLM:
├─ GPT-3.5-Turbo
└─ GPT-4

BERT系列:
├─ BERT-base
├─ BERT-large
└─ BERT-patent (专利领域微调)
```

**实验方法**:
1. **基于嵌入的方法**
   - 使用[CLS] token或最后token的隐藏状态
   - LoRA微调和全量微调
   - 纯文本和添加特征两种设置

2. **基于提示的方法**
   - Vanilla Prompt: 简单yes/no预测
   - CoT Prompt: 思维链逐步分析
   - 时间特征: 添加申请日期
   - Zero-shot, Few-shot, SFT

#### 3.4 实验结果: LLM的失败

**基于嵌入的结果** (Table 2):
| 模型 | AUC (纯文本) | AUC (特征) | Macro-F1 (特征) |
|------|-------------|------------|-----------------|
| BERT-base | 52.66 | 61.47 | 53.99 |
| BERT-large | 54.79 | 63.53 | 54.83 |
| **BERT-patent** | **55.81** | **63.63** | **54.91** |
| LLaMA-7B | 51.02 | 58.18 | 51.24 |
| LLaMA-70B | 52.11 | 60.44 | 53.46 |

**关键发现**:
- LLaMA-70B无法超越BERT-patent
- 简单扩展模型规模无效
- 特征工程比模型规模更重要
- 领域持续预训练有效 (BERT-patent最优)

**基于提示的结果** (Table 3):
| 模型 | Vanilla Prompt | +时间 | CoT Prompt | +时间 |
|------|----------------|-------|------------|-------|
| LLaMA-7B | 47.81 | 47.80 | 39.83 | 46.73 |
| Mistral-7B | 49.83 | 48.38 | 37.84 | 34.32 |
| Vicuna-7B | **49.83** | **49.43** | 37.84 | 44.23 |
| LLaMA-70B | 37.44 | **13.82** | 38.77 | 35.33 |
| GPT-3.5 | 48.38* | 48.81* | 23.93* | 10.27* |
| GPT-4 | 43.01* | 44.91* | 40.75* | 36.57* |

* 基于测试集子集 (1K样本)

**关键发现**:
- 最佳结果接近随机猜测 (50%)
- CoT Prompt反而损害性能
- 添加时间特征往往降低性能
- 大模型倾向于预测更多"no" (假阴性增加)

#### 3.5 FLAN Graph方法

**核心观察**: 专利权利要求存在两种依赖关系

```
权利要求依赖关系:

1. 内部权利要求依赖 (Inner-claim Dependency)
   ├─ 长权利要求内部是分层的
   ├─ 描述一个系统有多个(子)组件
   ├─ 例如: Claim 1描述系统有3个组件
   │             其中控制组件有4个功能
   └─ 识别方式: 冒号、分号分隔

2. 权利要求间依赖 (Inter-claim Dependency)
   ├─ 依赖权利要求引用其他权利要求
   ├─ 例如: Claim 2引用Claim 1
   ├─ 评估需要考虑祖先权利要求信息
   └─ 识别方式: 开头明确声明"X of claim Y"
```

**FLAN Graph构建** (Figure 3):

```
构建流程:

1. 节点构造
   ├─ 如果有内部依赖: 按标点和"专利用语"分解
   │   (comprising, consisting, whereby等)
   ├─ 每个节点代表一个(子)组件
   └─ 提取identities用于节点匹配

2. 边构造
   ├─ 内部依赖: 按分解时发现的层次连接
   ├─ 权利要求间依赖: 使用文本相似度匹配
   │   ├─ 使用StanfordCoreNLP进行词性标注
   │   ├─ 提取identities (锚点词/短语)
   │   ├─ 子权利要求的前言段匹配父图identities
   │   └─ 优先匹配最低级父identities
   └─ 方向: 从叶节点到根节点

3. 图表示
   ├─ 使用SentenceTransformer编码节点
   ├─ 使用GNN (GCN, GAT, GraphSage等)
   └─ 聚合根节点和目标节点作为最终表示
```

**FLAN Graph示例** (Figure 4):

```
Claim 2的FLAN Graph:

┌────────────────────────────────────┐
│  Claim 1 (继承)                    │
│  ├─ system                         │
│  ├─ authentication component       │
│  ├─ tracking component             │
│  └─ control component              │
│        ├─ receive auth info        │
│        ├─ receive location info    │
│        └─ deliver message          │
├────────────────────────────────────┤
│  Claim 2 (新增,红色背景)           │
│  └─ use location to recognize      │
└────────────────────────────────────┘

节点流向: 叶 → 根 (信息汇聚)
```

#### 3.6 实验结果: FLAN Graph的成功

**GNN模型结果** (Table 4):
| 模型 | AUC (纯FLAN) | AUC (特征) | Macro-F1 (特征) |
|------|-------------|------------|-----------------|
| GCN | 59.36 | **66.03** | 58.06 |
| GAT | 58.44 | 65.82 | 58.05 |
| GraphSage | 60.67 | **66.04** | **58.22** |
| TreeLSTM | 59.88 | 65.46 | 57.78 |

**vs SOTA (BERT-patent + 特征)**:
- GraphSage: AUC提升 **7.4%** (63.63→66.04)
- GraphSage: Macro-F1提升 **7.8%** (54.91→58.22)

**消融实验** (Table 5, Figure 6):
```
方法对比:

FLAN Graph (完整):
├─ GCN: 66.03 AUC, 58.06 Macro-F1
├─ GraphSage: 66.04 AUC, 58.22 Macro-F1
└─ 同时捕获内部和权利要求间依赖

Coarse Graph (仅权利要求间):
├─ GCN: 62.21 AUC, 54.69 Macro-F1
├─ GraphSage: 63.80 AUC, 56.64 Macro-F1
└─ 忽略内部依赖,性能下降

Solitary Node (无依赖):
├─ TreeLSTM: 60.17 AUC, 55.47 Macro-F1
├─ MLP: 59.33 AUC, 54.45 Macro-F1
└─ 完全忽略依赖,性能最低

结论: 两种依赖关系都对性能有贡献
```

**深度GNN实验** (Table 6):
- 将GNN从2层扩展到4层
- 性能没有明显提升
- 说明问题不是模型深度

### 📈 关键发现与数据

#### 4.1 LLM在专利任务上的局限性

**为什么LLM失败？**

1. **结构信息丢失**
   - LLM支持长上下文,但直接输入整个申请
   - 无法突出权利要求间的内部依赖结构
   - FLAN Graph显式捕获这些依赖

2. **时间敏感性挑战**
   - 新颖性取决于申请日期
   - LLM难以理解这种时间约束
   - 添加时间特征往往损害性能

3. **领域知识不足**
   - 需要技术和法律专业知识
   - 通用LLM缺乏专利领域深度理解
   - 领域微调 (BERT-patent) 更有效

4. **预测偏差**
   - 大模型倾向于预测更多"no"
   - 导致假阴性增加
   - 可能与训练数据分布有关

#### 4.2 特征工程的重要性

```
性能对比 (添加特征 vs 纯文本):

BERT-patent:
├─ 纯文本: 55.81 AUC
└─ +特征: 63.63 AUC (+7.82)

GraphSage + FLAN:
├─ 纯FLAN: 60.67 AUC
└─ +特征: 66.04 AUC (+5.37)

手工特征包括:
├─ 专利类别
├─ 引用数量
└─ 新颖性分数 (与5个最相关现有技术比较)
```

#### 4.3 知识驱动的价值

```
方法哲学对比:

LLM扩展方法:
└─ 相信更大的模型可以解决一切
   └─ 失败 ❌

知识驱动方法:
└─ 理解任务特点和领域知识
   └─ 成功 ✅

核心教训:
对于复杂的专业领域任务,
定制的领域特定方法
比简单扩展模型更有效
```

- 首个细粒度权利要求依赖图

**2. 全面的LLM评估**
- 测试7种不同大小的开源LLM
- 测试闭源GPT-3.5和GPT-4

- 多种配置: Zero-shot, Few-shot, SFT

**3. 关键发现**
- 揭示LLM在专利任务上的局限性
- 证明领域知识比模型规模更重要
- 为未来研究提供方向

**4. 开源贡献**
- 公开代码和数据集
- 提供详细实现细节
- 促进可重复研究

#### 5.2 技术优势

```
FLAN Graph优势:

1. 捕获结构信息
   ├─ 显式建模依赖关系

   ├─ 基于专利写作规范
```
