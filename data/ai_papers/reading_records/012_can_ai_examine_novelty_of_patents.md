### 🎯 核心内容

#### 3.1 专利审查流程解析

**实际审查过程**:
```
专利新颖性审查流程:

┌─────────────────────────────────────────────────────────┐
│                  专利审查流程图                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  [1] 提交专利申请                                        │
│         ↓                                               │
│  [2] 审查员检索现有技术 (Prior Art)                      │
│         ↓                                               │
│  [3] 对比权利要求与现有技术                              │
│         ↓                                               │
│  [4] 发现缺乏新颖性?                                     │
│         ├─ 是 → 发送 Non-Final Rejection                │
│         │         ↓                                    │
│         │    申请人修改权利要求                          │
│         │         ↓                                    │
│         │    重新对比 [3]                                │
│         │                                              │
│         └─ 否 → 批准专利                                │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**新颖性判断标准** (美国专利法35 U.S.C. §102):
> 如果在单一现有技术文献中找到权利要求发明的所有要素，则该发明被认为不具备新颖性。

#### 3.2 任务定义：Claim-Cited Texts (C-T) 输入

**创新任务设计**:
```
C-T 输入框架:

┌─────────────────────────────────────────────────────────┐
│                   二元分类任务                           │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  输入1: Patent Claim (权利要求)                         │
│    ├─ Non-Novel标签: 原始权利要求 (被驳回)              │
│    └─ Novel标签: 修改后权利要求 (获得授权)              │
│                                                         │
│  输入2: Cited Texts (引用的现有技术段落)                │
│    └─ 从Non-Final Rejection文档中提取                  │
│                                                         │
│  任务: 判断权利要求是否具备新颖性                        │
│    ├─ Novel: 权利要求要素未全部出现在现有技术中          │
│    └─ Non-Novel: 所有要素都在现有技术中找到             │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**数据来源设计**:
```
数据提取策略:

"Non-Novel" 数据:
├─ 来源: 收到Non-Final Rejection的专利申请
├─ Claim: 被驳回的原始权利要求
└─ Cited Texts: 驳回文档中引用的现有技术段落

"Novel" 数据:
├─ 来源: 同一专利申请的授权版本
├─ Claim: 申请人修改后的权利要求
└─ Cited Texts: 相同的现有技术段落

关键创新:
使用真实的Non-Final Rejection文档
确保数据与现实审查流程一致
```

#### 3.3 数据集构建

**数据来源**:
- USPTO APIs
- Google Patent Public Dataset
- Patent Examination Research Dataset
- Patent Application Office Actions数据

**数据规模**:
| 指标 | 数值 |
|------|------|
| 总专利数 | 3,975 |
| 训练:验证:测试 | 1,396:166:398 (预处理后) |
| 技术领域 | IPC G06F (电数字数据处理) |
| 时间范围 | 2014-2015年申请 |

**数据预处理**:
```
偏见消除策略:
├─ 问题: 较长的权利要求倾向于被标记为"Novel"
├─ 解决:
│   1. 按词数分组 (如100-110词)
│   2. 每组内随机过滤
│   3. 保持标签比例1:1
└─ 结果: 消除长度偏见,确保公平学习
```

### 📈 关键发现与数据

#### 4.1 实验设置

**测试模型**:
```
分类模型:
├─ Longformer-base-4096 (Encoder-only)
├─ Llama2 7B, 13B (Decoder-only)
└─ Llama3 8B, 70B (Decoder-only)

生成模型:
├─ Llama2 7B, 13B (指令微调版本)
├─ Llama3 8B, 70B (指令微调版本)
└─ GPT-4o

学习范式:
├─ Zero-shot
├─ Few-shot (2-shot)
└─ Supervised Fine-Tuning (SFT)

提示条件:
├─ Predict-Only: 只预测标签
└─ Explain-Predict: 先解释再预测
```

#### 4.2 人类基准测试

**人类准确率** (n=1, 20样本):
| 输入条件 | 准确率 | 说明 |
|----------|--------|------|
| Only Claim | 0.650 (13/20) | 仅凭权利要求文本 |
| Claim-Cited Texts | 0.600 (12/20) | 包含现有技术 |

**关键发现**:
- 仅凭权利要求可达到一定准确率 (基于技术内容和抽象程度)
- 添加现有技术后准确率未明显提升
- 任务难度较高,即使是人类专家也面临挑战

#### 4.3 主要实验结果

**分类模型结果** (Table 2):
| 模型 | Only Claim | Claim-Cited Texts |
|------|------------|-------------------|
| Longformer | 0.575 | 0.525 |
| Llama2 7B | 0.625 | 0.550 |
| Llama2 13B | 0.650 | 0.513 |
| Llama3 8B | 0.638 | 0.538 |

**关键发现**:
- Only Claim输入下准确率>0.65 (Llama3)
- C-T输入下性能接近0.5 (随机猜测水平)
- 引用文本可能作为噪声干扰分类

**生成模型结果** (Table 3):
```
Only Claim输入:
├─ Llama3 70B (few-shot): 0.700
├─ Llama3 70B (SFT): 0.688
├─ GPT-4o (few-shot): 0.675
└─ 超越人类基准 (0.650)

Claim-Cited Texts输入:
├─ Llama3 70B (Explain-Predict, few-shot): 0.625
├─ GPT-4o (Explain-Predict, few-shot): 0.600
└─ 大参数模型表现较好
```

#### 4.4 模型偏差分析

**Llama3 70B的预测偏差** (Table 4):
```
偏差特征:
├─ 倾向于预测"Novel"标签
├─ 对"Non-Novel"标签有高精确度
├─ Explain-Prompt下Non-Novel精确度更高

发现:
1. 生成模型确实评估C-T关系,而非仅依赖权利要求
2. 模型擅长识别差异,但不擅长评估相似性
3. 对明显对应关系的实例预测准确
```

#### 4.5 定性分析结果

**解释质量评估** (20个测试实例):

**成功案例特征**:
- 解释合理,涵盖每个权利要求要素
- 准确指出与现有技术的对应/差异部分
- 无幻觉生成 (hallucination)
- Non-Novel解释类似真实审查文档
- Novel解释突出修改后的新增部分

**失败案例原因**:
1. **复杂关系理解困难**
   - 上下位关系 (hypernyms/hyponyms)
   - 例子: "推测指令前缀" vs "潜在修改指令(LMI)"

2. **隐含关系识别失败**
   - 现有技术隐含相关细节而非明确陈述
   - 例子: 奇偶校验隐含在数据传输描述中

3. **专业知识不足**
   - 缺乏特定技术领域的深度知识
   - 难以理解复杂的技术细节

### 🚀 技术创新点

#### 5.1 核心创新

**1. 首个新颖性评估数据集**
- 基于真实专利审查案例
- 使用Non-Final Rejection文档
- Claim-Cited Texts配对结构

**2. C-T输入框架**
- 模拟真实审查流程
- 双输入条件设计
- 直接提取相关段落

**3. 多维度评估**
- 分类模型 vs 生成模型
- Zero-shot vs Few-shot vs SFT
- Predict-Only vs Explain-Predict

**4. 定性分析**
- 不仅看准确率,还分析解释质量
- 识别模型局限性
- 为未来改进提供方向

#### 5.2 重要发现

```
关键洞察:

1. 分类模型局限性
   ├─ 难以有效捕获Claim与Cited Texts的对应关系
   ├─ Cited Texts作为噪声而非有用信息
   └─ 需要新的任务设计方法

2. 生成模型潜力
   ├─ 大参数模型 (Llama3 70B) 达到合理准确率
   ├─ Explain-Prompt提供有价值的解释
   ├─ 超越人类基准在某些条件下
   └─ 但存在预测偏差

3. 任务挑战性
   ├─ 即使人类专家也面临困难
   ├─ 需要深刻理解技术内容和法律标准
   └─ 隐含关系和专业知识是主要障碍
```

### 📝 阅读笔记

**核心要点**:
1. 这是首个基于真实专利审查案例的新颖性评估数据集
2. 分类模型难以有效处理C-T输入任务
3. 大参数生成模型 (Llama3 70B, GPT-4o) 达到合理准确率
4. Explain-Predict生成的解释质量高,具有实用价值
5. 模型存在预测偏差,倾向于预测"Novel"

**技术亮点**:
- C-T输入框架巧妙模拟真实审查流程
- 使用Non-Final Rejection文档确保数据真实性
- 多维度评估揭示不同模型的特点
- 定性分析深入理解模型行为和局限性

**局限性**:
- 数据规模较小 (约1400训练样本)
- 仅覆盖G06F技术领域
- 模型对复杂关系理解有限
- 专业知识不足影响判断
- 提示工程探索有限

**待深入研究**:
- 中文专利的新颖性评估方法
- 更大规模和多领域的数据集
- 多模态信息 (附图) 的集成
- 专业知识注入方法
- 更优化的提示策略

**可借鉴思想**:
- C-T输入框架可用于其他对比任务
- Non-Final Rejection文档是宝贵数据源
- Explain-Predict提升可解释性和信任度
- 大参数模型在专业任务上的潜力
- 定性分析对理解模型行为的重要性
