### 阶段3: 上下文窗口优化（v2.1）

#### 时间：2025-12-26 下午

#### 问题识别：

**用户提问**: "这个提示词系统在使用时，会不会因为上下文太多而导致小娜的上下文窗口受限？"

**问题分析**:
- 通用模式提示词: ~63k tokens (49%占用)
- 加上检索结果和对话: ~85k+ tokens (66%+占用)
- 多轮对话空间严重不足
- 检索结果加载受限

#### 优化方案：

**1. 分层加载 (Layered Loading)**
- 按需加载提示词层级
- 基础层精简版: ~2k tokens (减少85%)
- 相关能力层: ~5k tokens
- 当前任务: ~3k tokens
- HITL精简版: ~1k tokens

**2. 能力按需 (Capability On-Demand)**
- 任务-能力映射系统
- 每个任务只加载需要的能力
- 例如: task_1_1只需要cap02_analysis（技术分析）

**3. 精简提示 (Prompt Compression)**
- 提取核心部分（标题、概述、执行流程）
- 压缩长段落（保留首尾关键句）
- 移除冗余示例
- 智能压缩算法

**4. 三种加载模式**

| 模式 | 预算Token | 适用场景 |
|-----|----------|---------|
| minimal | 4k | 快速问答、单步操作 |
| balanced | 12k | 标准任务（推荐默认） |
| full | 30k | 复杂任务、详细指导 |

#### 实现文件：

**优化版提示词加载器**
- 文件: `production/services/xiaona_prompt_loader_optimized.py`
- 核心功能:
  - `load_for_task(task_name)`: 为单个任务加载最小化提示词
  - `load_for_scenario(scenario)`: 为场景加载提示词
  - `_compress_prompt()`: 智能压缩算法
  - `_estimate_tokens()`: Token估算

**优化版小娜代理**
- 文件: `production/services/xiaona_agent_optimized.py`
- 核心功能:
  - 动态模式选择（根据上下文使用率自动调整）
  - 上下文使用追踪
  - 对话历史压缩
  - 下一步建议生成

#### 优化效果：

| 场景 | 原版 | 优化版(balanced) | 减少 |
|-----|------|-----------------|------|
| task_1_1 | 30k | 11k | **63%** |
| task_1_2 | 30k | 10k | **67%** |
| patent_writing | 30k | 11k | **63%** |
| office_action | 30k | 10k | **67%** |
| general | 63k | <1k | **98%** |

**上下文空间对比**:
```
原版: 系统提示63k + 用户2k + 检索20k = 85k (66%)
      剩余空间: 43k (34%)

优化版: 系统提示11k + 用户2k + 检索40k = 53k (41%)
        剩余空间: 75k (59%) ↑ 75%
```

**上下文优化方案文档**

  - 优化策略详解

### 阶段4: 自然语言交互界面

#### 时间：2025-12-26 下午晚些时候

#### 用户需求：

**用户提问**: "另外，我需要用自然语言使用该提示词系统"

#### 解决方案：

创建自然语言交互系统，支持直接用中文对话，无需编程。

#### 核心功能：

**1. 意图识别系统**
- 任务执行意图（"帮我分析这个技术交底书"）
- 场景切换意图（"切换到专利撰写模式"）
- 问题解答意图（"A26.4是什么要求？"）
- 控制指令意图（"状态"、"压缩对话历史"）

**2. 关键词映射**

任务关键词映射:
```
"理解技术交底|分析交底|交底书" → task_1_1
"检索现有技术|调研现有技术|对比分析" → task_1_2
"撰写说明书|写说明书" → task_1_3
...
```

场景关键词映射:
```
"专利撰写|撰写专利|写专利" → patent_writing
"意见答复|答复审查意见" → office_action
...
```

控制指令映射:
```
"状态|当前状态" → get_status
"压缩对话|压缩历史" → compress_history
"导出对话|保存对话" → export_conversation
...
```

**3. 智能模式检测**
- 用户可指定: "用最简模式快速检查"
- 系统自动选择: 根据上下文使用率动态调整
- 上下文>70% → minimal模式
- 上下文50-70% → balanced模式
- 上下文<50% → full模式

**4. 交互式对话循环**
- 输入"帮助"查看指令
- 输入"状态"查看使用情况
- 输入"退出"或Ctrl+C退出
- 支持多轮连续对话

#### 实现文件：

**自然语言交互界面**
- 文件: `production/services/xiaona_natural_interface.py`
- 核心类: `XiaonaNaturalInterface`
- 核心方法:
  - `chat(user_input)`: 处理用户输入
  - `_parse_intent(user_input)`: 解析用户意图
  - `_handle_task_execution()`: 处理任务执行
  - `_handle_scenario_switch()`: 处理场景切换
  - `_execute_control_command()`: 执行控制指令
  - `interactive_loop()`: 交互式对话循环

**快速启动脚本**
- 文件: `production/start_xiaona.sh`
- 功能: 一键启动自然语言交互系统
- 包含环境检查和错误处理

## 🎯 关键技术决策

### 1. 四层架构设计

**决策**: 采用分层架构而不是单一提示词

**理由**:
- 模块化：每层职责清晰
- 可扩展：新增能力或任务只需添加对应层文件
- 可重用：不同任务可以复用相同的能力
- 可维护：修改某层不影响其他层

**结果**:
- 10大能力可被9个任务灵活组合
- 新增场景只需定义业务层，能力层可直接复用

### 2. HITL机制设计

**决策**: 关键决策点必须由用户确认

**理由**:
- 专利法律工作需要专业判断
- AI建议需要人工审核
- 责任归属需要明确

**实现**:
- 每个任务包含3-5个交互点
- 标准化的交互模板
- 中断和回退机制

**结果**:
- 用户保持决策权
- AI提供建议和支持
- 符合专利代理实务

### 3. 上下文优化策略

**决策**: 采用分层加载+三种模式+智能压缩

**理由**:
- 原版系统占用66%上下文，无法支持长对话
- 不同任务对提示词完整性需求不同
- 需要平衡功能完整性和上下文空间

**实现**:
- 任务-能力映射：按需加载
- 三种模式：minimal(4k)/balanced(12k)/full(30k)
- 智能压缩：自动压缩超出预算的提示词
- 动态调整：根据上下文使用率自动选择模式

**结果**:
- Token减少65%
- 可用上下文提升75%
- 支持长多轮对话

### 4. 自然语言交互设计

**决策**: 支持自然语言而不是编程接口

**理由**:
- 目标用户是专利代理人，不一定是程序员
- 自然语言更直观易用
- 降低使用门槛

**实现**:
- 意图识别系统（关键词+正则表达式）
- 多种指令类型支持（任务、场景、问题、控制）
- 智能模式检测
- 交互式对话循环

**结果**:
- 无需编程即可使用
- 学习成本低
- 用户体验好
