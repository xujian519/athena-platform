Istio Auto Injection vs Manual Injection for Go Applications

目标
- 给出在 Go 应用场景中 Istio 自动注入与手动注入的权衡、成本与实施建议。

适用场景比较
- 自动注入 (Automatic Injection)
  - 优点：快速扩展、CI/CD 友好、统一代理配置，减少人工操作。
  - 缺点：对现有的部署模板依赖性强，排错在初期可能较复杂；对资源消耗的微管理能力相对较低。
- 手动注入 (Manual Injection via istioctl kube-inject 或注入模板)
  - 优点：对边车配置有更明确控制，能在对成本敏感场景中进行定制化注入。
  - 缺点：部署复杂度高，CI/CD 流程需要额外支持；版本不一致可能导致问题。

影响因素
- 服务数量与迭代速度：小型/中型服务集群倾向自动注入，大型分布式系统可能需要手动注入以控制每个服务的副本与资源。
- dev/prod 环境一致性：自动注入有助于环境一致性，手动注入需额外的治理。
- 安全与合规：对身份、证书、日志审计的集中化治理影响自动化程度。

实施成本对比
- 自动注入：初始成本较低，长期运维成本下降。
- 手动注入：初期工作量大，自动化程度不高时成本上升。

落地建议
- 新项目或中小规模微服务集群优先考虑自动注入，确保命名空间标签和命名约束正确。
- 对于关键/旧系统、需要严格自定义侧车行为的服务，采用手动注入并逐步合并到统一策略。
- 在 CI/CD 流程中统一注入策略，避免不同环境下注入行为差异。

示例与要点
- 标准命名空间标签: istio-injection=enabled/disabled
- 注入验证：kubectl get deployments -o yaml 中的 initContainers/sidecar 注入情况
- 监控与回滚：确保注入版本变更时有观测和回滚点。

结论
- 对于 Athena Gateway 的网格迁移，初期推荐使用 Istio 自动注入以提升效率与可观测性，必要时对个别服务切换到手动注入以实现细粒度控制。
