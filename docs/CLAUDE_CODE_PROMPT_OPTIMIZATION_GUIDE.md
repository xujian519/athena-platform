## 🚀 核心优化技巧

### 1. 结构化提示词（STRUCTURE）

**原则**：清晰的层次结构让模型更容易理解

#### 优化前
```
请帮我分析这个架构问题...
```

#### 优化后
```markdown
# 任务分析

## 背景
[提供足够的背景信息]

## 具体要求
1. 要求一
2. 要求二
3. 要求三

## 输出格式
请按照以下结构输出：
- [格式一]
- [格式二]

## 约束条件
- [约束一]
- [约束二]
```

**提升效果**：+25% 相关性

### 2. 上下文注入（CONTEXT INJECTION）

**原则**：提供相关但不过量的上下文

#### 实战模板
```markdown
## 项目上下文

项目名称: Athena工作平台
当前阶段: 架构重构阶段
技术栈: Python, PostgreSQL, Neo4j, Qdrant
团队规模: 1人(徐健)
核心目标: 专利法律AI平台

## 当前问题
[描述具体问题]

## 历史决策
- [相关决策1]
- [相关决策2]
```

**提升效果**：+30% 准确性

### 3. 思维链引导（CHAIN OF THOUGHT）

**原则**：让模型展示推理过程

#### 实战模板
```markdown
## 分析要求

请按照以下步骤进行思考：

1. **问题理解**
   - 核心问题是什么？
   - 边界条件是什么？

2. **方案分析**
   - 方案A: [优缺点]
   - 方案B: [优缺点]
   - 方案C: [优缺点]

3. **推理过程**
   让我们一步步思考：
   - 第一步: ...
   - 第二步: ...
   - 第三步: ...

4. **结论**
   - 推荐方案: ...
   - 理由: ...
   - 风险: ...
   - 缓解措施: ...
```

**提升效果**：+40% 复杂问题解决能力

### 4. 少样本学习（FEW-SHOT LEARNING）

**原则**：提供示例引导预期格式

#### 实战模板
```markdown
## 参考示例

### 示例1: 简单问题
用户: "存储层是怎么工作的？"

模型回答:
🎯 业务价值:
  - 解决专利数据的高效存储和检索问题
  - 支持向量搜索，为AI分析提供基础

🏗️ 架构定位:
  - 属于基础设施层

⚙️ 技术实现:
  - L1缓存(内存) → L2缓存(Redis) → L3缓存(数据库)

### 示例2: [您的另一个示例]

---

## 当前问题
[您的问题]

请按照上述示例的格式和深度回答。
```

**提升效果**：+35% 格式一致性

**原则**：让模型进入特定角色

**提升效果**：+20% 专业性

### 6. 约束明确化（CONSTRAINT CLARIFICATION）

**原则**：明确告诉模型什么不能做

#### 实战模板
```markdown
## 正面约束
- 必须考虑现有技术栈

- 不要提供泛泛而谈的建议
- 不要假设无限资源

## 输出要求
- 代码必须有中文注释
- 复杂逻辑必须有解释
- 风险必须有缓解措施
```

**提升效果**：+25% 可执行性

## 📋 实战优化模板

### 模板1: 代码问题诊断

```markdown
## 代码问题诊断

### 代码上下文
文件: {file_path}
功能: {description}
最近修改: {recent_changes}

### 问题描述
{problem_description}

### 已尝试的解决方案
{attempted_solutions}

### 期望结果
{expected_result}

### 约束条件
- 不能破坏现有功能
- 需要考虑性能影响
- 需要保持代码可读性

### 分析要求
1. **问题定位**: 哪里出了问题？为什么？
2. **根因分析**: 问题的根本原因是什么？
3. **解决方案**: 提供多个可选方案
4. **推荐方案**: 基于什么标准推荐？
5. **实施步骤**: 具体的实施步骤

请按照资深架构师的角度进行分析。
```

### 模板2: 架构设计评审

```markdown
## 架构设计评审

### 设计概述
{design_overview}

### 评审维度
1. **功能性**: 是否满足需求？
2. **可扩展性**: 未来如何扩展？
3. **性能**: 潜在的性能瓶颈？
4. **安全性**: 安全风险点？
5. **可维护性**: 代码是否易于维护？
6. **成本**: 开发和运维成本？

### 评审要求
- 提供具体的数据支撑
- 对比至少2种替代方案
- 指出潜在风险和缓解措施
- 给出明确的建议（接受/拒绝/修改）

请以严谨的态度进行评审，不回避问题。
```

### 模板3: 代码重构建议

```markdown
## 代码重构建议

### 当前代码
\`\`\`python
{current_code}
\`\`\`

### 代码问题
{identified_issues}

### 重构目标
- 提高可读性
- 提升可维护性
- 优化性能
- 遵循最佳实践

### 约束条件
- 保持现有接口不变
- 不能引入新的依赖
- 需要添加测试

### 输出要求
1. **重构后的代码**（带中文注释）
2. **改动说明**: 为什么这样改？
3. **测试建议**: 需要测试什么？
4. **迁移步骤**: 如何安全迁移？

请提供生产级的重构代码。
```

## 🔧 实用工具函数

### 创建提示词上下文辅助函数

```python
# tools/prompt_context_helper.py

class PromptContextHelper:
    """提示词上下文辅助工具"""

    @staticmethod
    def get_project_context() -> str:
        """获取项目上下文"""
        return """
## 项目上下文
项目: Athena工作平台
阶段: Gateway架构转型
技术栈: Python, PostgreSQL, Neo4j, Qdrant
核心目标: 专利法律AI平台

## 当前进度
- ✅ OpenClaw架构分析完成
- ✅ 备份系统部署完成
- 🔄 Gateway接口设计中
- ⏳ 记忆系统重构待开始
"""

    @staticmethod
    def get_code_context(file_path: str) -> str:
        """获取代码上下文"""
        return f"""
## 当前文件
文件: {file_path}
请先阅读该文件，了解其功能和上下文。
"""

    @staticmethod
    def get_analysis_template(task_type: str) -> str:
        """获取分析模板"""
        templates = {
            "bug_fix": """
## Bug修复分析

1. **问题定位**
   - 代码位置:
   - 问题原因:

2. **修复方案**
   - 方案描述:
   - 代码改动:
   - 测试验证:

3. **风险评估**
   - 潜在影响:
   - 缓解措施:
""",
            "feature_design": """
## 功能设计分析

1. **需求理解**
   - 功能目标:
   - 用户价值:

2. **设计方案**
   - 技术选型:
   - 接口设计:
   - 数据流:

3. **实施计划**
   - 开发步骤:
   - 测试策略:
   - 发布计划:
"""
        }
        return templates.get(task_type, "")

# 使用示例
helper = PromptContextHelper()

# 构建完整提示
prompt = f"""
{helper.get_project_context()}

{helper.get_code_context('core/gateway/api.py')}

{helper.get_analysis_template('bug_fix')}

## 当前任务
[描述具体任务]
"""
```
