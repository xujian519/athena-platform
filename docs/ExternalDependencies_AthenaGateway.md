External Dependencies Compatibility for Athena Gateway in Mesh

目标
- 明确 Athena Gateway 在服务网格（Istio/Linkerd）环境中的外部依赖需求、兼容性与落地策略。

范围
- DNS 解析、证书管理、外部鉴权、以及 API 密钥等外部依赖。

现状清单
- DNS：外部域名解析、跨命名空间服务发现、跨集群访问。
- 证书管理：证书轮换、TLS 配置、证书签发机构信任链管理。
- 外部鉴权：OIDC/JWT、OAuth、API 网关令牌等。
- API 密钥：在入口层或应用层的密钥校验机制，以及日志审计。

网格中的潜在处理模式
- DNS 与服务发现：Mesh 提供服务网格内部的 DNS 解析，外部 DNS 需要与集群外部的名称解析保持一致，必要时通过入口网关进行转发策略。
- 证书管理：通过 Istio SDS/秘密管理集成证书轮换，或通过 Cert-Manager 集成外部 CA 证书在网格内分发。
- 外部鉴权：将外部鉴权移至网格内的策略层（AuthorizationPolicy、RequestAuthentication）或通过外部认证服务与网格对接。
- API 密钥：可在网关层进行初步校验，或通过基于策略的服务间调用实现更细颗粒度控制。

兼容性分析要点
- 安全性： mesh 化通常提升密钥、证书与身份管理的一致性与审计能力。
- 延迟/复杂度：网格引入的额外跳数和 TLS 握手可能带来微小的延迟增加，需通过基准测试量化。
- 自动化运维： cert-manager、秘密密钥轮换、策略授权的自动化程度提升，对运维友好。
- 兼容性风险：某些外部系统对客户端证书、IP 白名单等有严格要求，需要在网格中保留对外访问路径或提供等效能力。

落地建议
- 证书管理：在网格环境中统一使用证书管理工具（如 cert-manager + Istio SDS），实现证书轮换的自动化。
- DNS/外部访问：将外部入口域名与网格内部服务名称映射，通过 Gateway/VirtualService 进行跨命名空间访问控制。
- 外部鉴权：引入网格级鉴权策略，必要时接入外部身份提供者（OIDC）并在 RequestAuthentication 层实现令牌校验。
- API 密钥：在网关层实现统一校验/审计，或在网格内引入服务级别的密钥策略，以便追踪与合规性。
- 验证计划：构建 PoC 测试用例，覆盖 DNS 解析、证书轮换、鉴权流程、API 密钥校验，以及跨命名空间的访问。

风险与缓解
- 风险：证书轮换失败、鉴权对接失败、外部系统不可用导致网关阻塞。
- 缓解：实现回滚点、健康检查、预备证书、分阶段切换、完整监控告警。
