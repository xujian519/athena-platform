## 2. 高风险项详解

### 🔴 风险 #1: 破坏性变更（Breaking Changes）

**风险描述**:
- 6个注册表系统API不兼容
- 统一后可能导致现有代码失效
- 影响394个文件

**影响范围**:
- 直接影响: 67个文件（ToolRegistry使用者）
- 间接影响: 327个文件（其他注册表使用者）
- 总计: 约394个文件

**概率**: 🔴 **高** (80%)

**影响**: 🔴 **严重** (系统不可用)

**缓解措施**:
1. **兼容性层**:
   ```python
   # 保留旧API的适配器
   class ToolRegistryCenterAdapter:
       def __init__(self):
           self._new_registry = get_global_registry()

       def register_tool(self, tool_info):
           # 转换为新的ToolDefinition
           return self._new_registry.register(...)

       # 其他适配方法...
   ```

2. **渐进式迁移**:
   - Phase 1: 新旧系统共存
   - Phase 2: 逐步迁移代码
   - Phase 3: 移除旧系统

3. **版本控制**:
   - 使用语义化版本号
   - 明确标记breaking changes
   - 提供迁移指南

**应急预案**:
1. Git分支管理（保留旧系统分支）
2. 快速回滚脚本
3. 数据库备份
4. 灰度发布

### 🔴 风险 #2: 循环依赖（Circular Dependencies）

**风险描述**:
- 整合过程中可能引入循环依赖
- 2处潜在风险已识别
- 可能导致模块加载失败

**影响范围**:
- 核心工具系统
- 搜索系统
- 治理系统

**概率**: 🔴 **高** (70%)

**影响**: 🔴 **严重** (系统崩溃)

**缓解措施**:
1. **依赖注入**:
   ```python
   # 使用依赖注入打破循环
   class ToolManager:
       def __init__(self, registry: ToolRegistry):
           self._registry = registry

   # 延迟注入，而非import时依赖
   manager = ToolManager(registry=get_global_registry())
   ```

2. **接口抽象**:
   ```python
   # 定义抽象接口
   class IToolRegistry(ABC):
       @abstractmethod
       def get_tool(self, tool_id: str) -> ToolDefinition | None:
           pass

   # 所有模块依赖接口，不依赖实现
   ```

3. **延迟加载**:
   ```python
   # 使用lazy import
   def get_registry():
       from core.tools.base import get_global_registry
       return get_global_registry()
   ```

4. **依赖检测**:
   - 使用工具（pydeps）自动检测循环依赖
   - CI/CD集成检测
   - 禁止循环依赖的代码审查规则

**应急预案**:
1. 模块重构（打破循环）
2. 代码降级（恢复到无循环版本）
3. 紧急补丁（临时解决循环）

- 缓存策略可能需要重新设计
- 响应时间可能增加

**影响范围**:
- 工具查询性能
- 工具调用性能
- 系统整体性能

**概率**: 🟡 **中** (50%)

**影响**: 🔴 **严重** (用户体验下降)

**当前性能基线**:
- 工具查询: < 10ms (LRU缓存)
- 工具调用: < 100ms (P95)
- 缓存命中率: 85%

**缓解措施**:
1. **性能测试**:
   ```python
   # 建立性能基准
   benchmarks = {
       "find_by_category": 10,  # ms
       "find_by_domain": 10,    # ms
       "search_tools": 20,      # ms
   }

   # 回归测试
   def test_performance_regression():
       assert measure_time(find_by_category) < benchmarks["find_by_category"]
   ```

2. **缓存优化**:
   - 保留LRU缓存（128/256项）
   - 增加缓存预热
   - 优化缓存失效策略

3. **性能监控**:
   - 实时监控响应时间
   - 自动告警（超过阈值）
   - 性能趋势分析

**应急预案**:
1. 回退到旧系统
2. 禁用性能差的特性
3. 横向扩展（增加实例）

### 🔴 风险 #4: 数据不一致（Data Inconsistency）

**风险描述**:
- 多个注册表状态可能不一致
- 工具定义可能冲突
- 缓存数据可能过期

**影响范围**:
- 工具状态管理
- 性能统计
- 健康检查

**概率**: 🟡 **中** (60%)

**影响**: 🔴 **严重** (决策错误)

**缓解措施**:
1. **单一数据源**:
   - 只保留一个主注册表
   - 其他作为只读副本
   - 定期同步状态

2. **版本控制**:
   ```python
   @dataclass
   class ToolDefinition:
       version: str = "1.0.0"
       updated_at: datetime = field(default_factory=datetime.now)

       def is_newer_than(self, other: ToolDefinition) -> bool:
           return self.updated_at > other.updated_at
   ```

3. **一致性检查**:
   ```python
   def check_consistency(registries: list[ToolRegistry]):
       # 检查工具定义是否一致
       tools_by_id = defaultdict(list)
       for registry in registries:
           for tool in registry.get_all_tools():
               tools_by_id[tool.tool_id].append(tool)

       # 检查冲突
       conflicts = []
       for tool_id, definitions in tools_by_id.items():
           if len(definitions) > 1:
               conflicts.append((tool_id, definitions))

       return conflicts
   ```

**应急预案**:
1. 数据库备份恢复
2. 强制重新同步
3. 人工介入修复
