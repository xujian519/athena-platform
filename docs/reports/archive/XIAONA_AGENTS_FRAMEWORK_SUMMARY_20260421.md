> **阶段**: Phase 1 核心框架实现

> **实施者**: Claude Code + 徐健

## 📊 项目概览

### 目标

将小娜智能体拆分为多个专业智能体，通过小诺的动态编排实现灵活组合，提升系统的专业性和可维护性。

### 实施策略

- ✅ **渐进式拆分**: 分3个阶段逐步实现
- ✅ **向后兼容**: 保留小娜统一入口
- ✅ **动态编排**: 小诺根据场景自动组装智能体
- ✅ **易于扩展**: 新增智能体只需注册

## 📁 已创建文件清单

### 1. 核心框架代码

#### 智能体基础组件
```
core/agents/xiaona/
├── __init__.py                      # 模块初始化，导出Phase 1智能体
└── base_component.py                # 基础组件类，定义统一接口
```

**关键特性**:
- `BaseXiaonaComponent`: 所有专业智能体的基类
- `AgentCapability`: 能力描述数据类
- `AgentExecutionContext`: 执行上下文
- `AgentExecutionResult`: 执行结果
- 生命周期管理: `_initialize()` → `execute()` → 结果返回

#### Phase 1 专业智能体
```
core/agents/xiaona/
├── retriever_agent.py               # 检索者（专利检索专家）
├── analyzer_agent.py                # 分析者（技术分析专家）
└── writer_agent.py                  # 撰写者（文书撰写专家）
```

**智能体能力矩阵**:

| 智能体 | 核心能力 | 输入 | 输出 | 预估时间 |
|--------|---------|------|------|---------|
| 检索者 | patent_search, keyword_expansion, document_filtering | 查询关键词 | 专利列表 | 15s |
| 分析者 | feature_extraction, novelty_analysis, creativity_analysis | 专利文本 | 分析报告 | 20-30s |
| 撰写者 | claim_drafting, description_drafting, office_action_response | 技术特征 | 申请文件 | 30-90s |

#### 编排系统
```
core/orchestration/
├── __init__.py                      # 模块初始化
├── agent_registry.py                # 智能体注册表（单例模式）
├── scenario_detector.py             # 场景识别器
├── workflow_builder.py              # 工作流构建器
└── execution_monitor.py             # 执行监控器
```

**核心组件说明**:

1. **AgentRegistry** (智能体注册表)
   - 管理所有智能体的注册、查询、发现
   - 线程安全，支持智能体的启用/禁用
   - 提供按能力、按阶段的查询接口

2. **ScenarioDetector** (场景识别器)
   - 支持7种业务场景自动识别
   - 关键词匹配 + 场景配置
   - 返回需要的智能体组合

3. **WorkflowBuilder** (工作流构建器)
   - 根据场景构建工作流
   - 支持4种执行模式：串行、并行、迭代、混合
   - 自动处理步骤依赖关系

4. **ExecutionMonitor** (执行监控器)
   - 实时监控工作流执行状态
   - 支持断点续传和错误恢复
   - 提供进度查询接口

#### 小诺编排者
```
core/agents/
└── xiaonuo_orchestrator.py          # 小诺·智能体编排者（增强版）
```

**核心职责**:
1. 场景识别：根据用户输入识别业务场景
2. 智能体组装：根据场景选择需要的智能体
3. 工作流构建：构建智能体执行工作流
4. 执行监控：监控工作流执行状态
5. 结果聚合：聚合各智能体的执行结果

## 🎯 技术亮点

### 1. 渐进式架构设计

```
Phase 1 (当前): 3个核心智能体
    ↓
Phase 2: +2个专业智能体（规划者、规则官）
    ↓
Phase 3: +2个辅助智能体（润色师、翻译官）
```

**优势**:
- 每个阶段都可以独立运行和验证
- 降低重构风险
- 便于A/B测试和数据驱动决策

### 2. 动态编排机制

```
用户请求 → 小诺场景识别 → 智能体组合确定 → 工作流生成 → 执行监控
```

**场景映射示例**:
- 专利检索 → [检索者]
- 创造性分析 → [规划者 → 检索者 → 分析者 → 规则官]
- 专利撰写 → [规划者+检索者(并行) → 分析者 → 规则官 → 撰写者 → 润色师]

### 3. 统一的基础设施

- ✅ 智能体注册表（类似统一工具注册表）
- ✅ 统一的执行上下文和结果格式
- ✅ 统一的能力描述机制
- ✅ 统一的生命周期管理

### 4. 完善的监控和错误处理

- ✅ 执行状态实时监控
- ✅ 错误恢复和断点续传
- ✅ 详细的日志记录
- ✅ 性能指标收集

## 📈 与现有架构的对比

| 维度 | 拆分前 | 拆分后（Phase 1） | 改进 |
|------|--------|------------------|------|
| 智能体数量 | 1个大而全的小娜 | 3个专业小娜 + 1个小诺 | 专业性提升 |

| 职责分离 | 模糊 | 清晰（SRP） | 可测试性↑ |

| 场景识别 | 规则匹配 | 智能识别 | 准确率↑ |
