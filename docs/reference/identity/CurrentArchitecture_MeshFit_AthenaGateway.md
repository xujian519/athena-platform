Current Architecture Fit for Mesh - Athena Gateway

目标
- 收集、整理并文档化 Athena Gateway 现有架构的拓扑、路由、TLS 终止、鉴权、以及性能/扩展约束，为后续服务网格集成提供清晰的起点与风险点。

现状摘要
- Athena Gateway 作为对外 API 的入口网关，处理入口 TLS、认证鉴权、路由分发，以及对后端服务的代理调用。当前环境中，网关与后端微服务的部署、证书管理和流量控制依赖于传统的反向代理/网关配置，尚未统一进入服务网格。

拓扑概览（简化示意）

Ingress/边缘入口 ── Hera/Athena Gateway ── 后端微服务集群
                      │
                      ├─ 服务发现与证书管理（当前实现）
                      └─ 鉴权/鉴证与路由策略（当前实现）

关键组件对照
- TLS/证书：由边缘终止点或网关代理执行 TLS，证书轮换比较分散，缺乏统一的证书管理策略。
- 路由/转发：基于路径、方法与参数的路由表，缺乏一致的全局路由策略与跨服务路由的可观测性。
- 鉴权：可能集中在网关层实现，缺乏细粒度的服务间授权。
- 观测：监控/追踪通常分散在网关与后端服务，跨服务端到端的可观测性有限。
- 性能/扩展性：高并发时段可能出现瓶颈，现有扩展手段多依赖手动调整，缺乏网格化的弹性伸缩能力。

路由与流量点
- 入站流量通过网关接入，网关再将请求路由到后端微服务或聚合层。
- 路由配置通常存在多份副本、分布在不同环境（开发/测试/生产），缺少统一的版本管控。

TLS 终止与证书管理
- 入口 TLS 终止位置不统一，证书轮换流程不一致，存在潜在的安全与合规风险。
- 若启用端到端 TLS，需要在网关与后端服务之间维持加密通道，增加配置复杂性。

鉴权与策略
- 鉴权策略多样，跨命名空间的策略复制成本高，难以在全栈统一来回放。
- 现有策略对服务间调用的保护较弱，缺少细粒度的策略执行与审计。

性能与扩展约束
- 基线延迟与并发能力需明确量化，现有监控粒度不足以快速定位瓶颈。
- 网关的扩展性受限于手工扩容、静态路由表和非统一的资源配额管理。

兼容性评估要点
- 兼容性： mesh 原语（sidecar、Gateway、VirtualService、DestinationRule、mTLS、Policy 等）是否能无缝对接现有路由与鉴权模型。
- 变更成本：证书管理、鉴权模型、路由语义的变更成本与培训成本。
- 风险点：端到端安全策略、观测体系迁移、回滚策略等的复杂性。

下一步建议
- 产出 CurrentArchitecture_MeshFit_Risks.md 与 CurrentArchitecture_MeshFit_ChangeLog.md，列出风险、缓解措施、以及变更版本、时间表。
- 为 1.2-1.4 的任务准备输入：网格映射、外部依赖兼容性、以及变更清单。

变更清单（草案）
- 引入网格化的证书管理与 TLS 策略，统一证书轮换机制。
- 采用统一的服务发现与路由策略模板，逐步迁移至网格化路由语义。
- 设计端到端加密与分段解密的策略。
