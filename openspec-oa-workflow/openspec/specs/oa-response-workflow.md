# 审查意见答复工作流规范 v2.0

**版本**: 2.0.0  
**创建日期**: 2026-03-26  
**更新日期**: 2026-03-26  
**适用范围**: 专利申请审查意见答复全流程

## 1. 工作流总览

```
┌─────────────────────────────────────────────────────────────┐
│  Step 1: 接收指令 + 解析审查意见                              │
│  ├─ 确认审查意见通知书存在（PDF/图片格式）                    │
│  └─ 使用JSON模板解析审查意见                                  │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  Step 2: 材料收集                                            │
│  ├─ 确认对比文件（审查员引用的D1/D2...）                      │
│  ├─ 确认原始申请文件（PDF/DOC/DOCX + 附图）                   │
│  ├─ [可选] 根据公开号自动下载文件                             │
│  └─ 材料不齐 → 返回提示，请求补充                             │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  Step 3: 双智能体并行分析                                    │
│  ┌─────────────────────┐    ┌─────────────────────┐         │
│  │ 子智能体1            │    │ 子智能体2            │         │
│  │ ├─ 审查意见深度分析  │    │ ├─ 对比文件1分析    │         │
│  │ └─ 原始申请文件分析  │    │ ├─ 对比文件2分析    │         │
│  │                     │    │ └─ ...              │         │
│  │ 输出: JSON + MD     │    │ 输出: JSON + MD     │         │
│  └─────────────────────┘    └─────────────────────┘         │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  Step 4: 主智能体制定策略                                    │
│  ├─ 简单意见 → 使用法条直接分析                              │
│  ├─ 复杂意见 → 法律世界模型（考虑技术领域）                   │
│  └─ ⚠️ 必须用户确认后才能执行                                │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  Step 5: 撰写答复（子智能体 + 逐项确认）                      │
│  ├─ 先写提纲 → 用户确认                                     │
│  └─ 按提纲逐项撰写 → 每完成一项 → 用户确认 → 下一项           │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  Step 6: 双智能体辩论验证                                    │
│  ┌─────────────────────┐    ┌─────────────────────┐         │
│  │ 🎭 代理师角色        │ ←→ │ 🔍 审查员角色        │         │
│  │ 捍卫答复观点        │    │ 挑战答复漏洞        │         │
│  └─────────────────────┘    └─────────────────────┘         │
│                      ↓                                      │
│                ⚖️ 主智能体居中裁判                            │
│                规则: ≥5轮 或 达成一致                         │
│                要求: 必须真实LLM支持                          │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  Step 7: 输出文件 + 用户最终确认                              │
│  ├─ 输出辩论记录                                            │
│  ├─ 根据辩论结果修改答复                                    │
│  ├─ 生成最终文件包                                          │
│  └─ 用户最终确认                                            │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  Step 8: 自我反思与持续改进                                  │
│  ├─ 对照用户的修改内容                                       │
│  ├─ 分析修改原因                                            │
│  ├─ 反思：为什么没有考虑到                                   │
│  ├─ 改进建议：更新答复流程/模板                              │
│  └─ 请用户确认是否采纳改进                                   │
└─────────────────────────────────────────────────────────────┘
```

## 2. Step 1: 接收指令与解析

### 2.1 输入要求

系统 **SHALL** 接收以下输入：

| 输入项 | 格式 | 必需性 | 说明 |
|--------|------|--------|------|
| 审查意见通知书 | PDF/图片 | **必需** | 国知局发出的官方文件 |
| 明确指令 | 文本 | **必需** | 用户明确要求答复审查意见 |

### 2.2 解析规范

系统 **SHALL** 使用JSON模板解析审查意见：

```json
{
  "notification_info": {
    "notification_number": "通知书编号",
    "mail_date": "发文日",
    "deadline": "答复期限"
  },
  "application_info": {
    "application_number": "申请号",
    "applicant": "申请人",
    "invention_title": "发明名称",
    "filing_date": "申请日"
  },
  "examination_opinions": [
    {
      "opinion_id": "意见序号",
      "claim_affected": "涉及的权利要求",
      "opinion_type": "新颖性|创造性|不清楚|不充分|不支持|其他",
      "legal_basis": "法条依据",
      "examiner_reasoning": "审查员理由",
      "cited_references": [
        {
          "reference_id": "D1/D2/...",
          "document_number": "文献号",
          "relevant_paragraphs": "相关段落"
        }
      ]
    }
  ],
  "examiner_suggestions": "审查员建议",
  "other_issues": ["其他问题"]
}
```

## 3. Step 2: 材料收集

### 3.1 必需材料

系统 **SHALL** 确认以下材料：

| 材料 | 格式 | 说明 |
|------|------|------|
| 对比文件 | PDF | 审查员引用的D1、D2... |
| 原始申请文件 | PDF/DOC/DOCX | 包含说明书、权利要求书、附图 |

### 3.2 自动下载（可选）

当用户提供公开号时，系统 **SHOULD** 自动下载：
- 原始申请文件的公开文本
- 对比文件全文

### 3.3 材料缺失处理

系统 **SHALL** 在材料不齐时：
1. 明确告知用户缺少哪些材料
2. 说明这些材料的用途
3. 提供获取材料的建议

---

## 4. Step 3: 双智能体并行分析

### 4.1 子智能体1：审查意见 + 申请文件分析

系统 **SHALL** 启动子智能体1执行：

| 分析对象 | 输出格式 | 内容 |
|----------|----------|------|
| 审查意见 | JSON + MD | 审查员核心论点、论证逻辑、法条依据 |
| 原始申请 | JSON + MD | 技术方案、发明点、技术效果、权利要求保护范围 |

### 4.2 子智能体2：对比文件分析

系统 **SHALL** 启动子智能体2执行：

| 分析对象 | 输出格式 | 内容 |
|----------|----------|------|
| 对比文件1 | JSON + MD | 技术方案、公开内容、技术领域 |
| 对比文件2 | JSON + MD | 技术方案、公开内容、技术领域 |
| ... | ... | ... |

### 4.3 分析输出规范

每个分析 **SHALL** 输出：
- **JSON文件**：结构化数据，便于程序处理
- **Markdown文件**：可读性强的文档，便于人工审核

---

## 5. Step 4: 策略制定

### 5.1 策略选择规则

系统 **SHALL** 根据意见复杂程度选择策略：

| 复杂程度 | 判断标准 | 策略工具 |
|----------|----------|----------|
| 简单 | 单一问题、法条明确 | 法条直接分析 |
| 复杂 | 多问题交叉、技术领域特殊 | 法律世界模型 |

### 5.2 法律世界模型要求

使用法律世界模型时，系统 **SHALL**：
1. 考虑专利的技术领域
2. 参考该领域的审查惯例
3. 结合具体技术特征分析

### 5.3 用户确认机制

系统 **SHALL** 在执行前：
1. 展示制定的答复策略
2. 说明选择该策略的理由
3. 等待用户明确确认
4. 用户可提出修改意见

---

## 6. Step 5: 撰写答复

### 6.1 提纲优先

系统 **SHALL** 先生成答复提纲：
1. 审查意见概述
2. 各项答复的结构
3. 预计论述要点

用户确认提纲后才能继续。

### 6.2 逐项撰写与确认

系统 **SHALL** 按以下流程撰写：

```
提纲确认 → 撰写第1项 → 用户确认 → 撰写第2项 → 用户确认 → ... → 完成
```

每项撰写后 **SHALL**：
1. 展示已完成内容
2. 等待用户确认
3. 根据反馈修改
4. 确认无误后进入下一项

---

## 7. Step 6: 双智能体辩论验证

### 7.1 角色设定

系统 **SHALL** 配置两个子智能体：

| 角色 | 职责 | 立场 |
|------|------|------|
| 🎭 代理师 | 捍卫答复观点 | 支持答复的合理性 |
| 🔍 审查员 | 挑战答复漏洞 | 质疑答复的不足 |

### 7.2 辩论规则

系统 **SHALL** 执行以下规则：

| 规则 | 要求 |
|------|------|
| 最少轮数 | ≥ 5轮 |
| 结束条件 | 达成一致 或 轮数达标 |
| LLM要求 | 必须真实LLM支持，不能模拟 |
| 辩论内容 | 针对答复的每个论点 |

### 7.3 主智能体裁判

主智能体 **SHALL**：
1. 监控辩论进程
2. 记录争议点
3. 在争议无法达成一致时裁决
4. 总结辩论结论

### 7.4 辩论输出

系统 **SHALL** 输出：
- 完整辩论记录（Markdown格式）
- 争议点列表
- 最终结论

---

## 8. Step 7: 输出与最终确认

### 8.1 根据辩论修改

系统 **SHALL**：
1. 根据辩论中暴露的问题修改答复
2. 补充论据
3. 调整表述

### 8.2 最终文件包

系统 **SHALL** 生成以下文件：

```
答复包/
├── 意见陈述书.docx              # 提交给国知局的主文件
├── 修改对照表.docx              # 权利要求修改对照（如有修改）
├── 辩论记录.md                  # 双智能体辩论完整记录
├── 答复策略.md                  # 答复策略说明
└── 分析文件/
    ├── 审查意见分析.json
    ├── 审查意见分析.md
    ├── 申请文件分析.json
    ├── 申请文件分析.md
    ├── 对比文件1分析.json
    ├── 对比文件1分析.md
    ├── 对比文件2分析.json
    ├── 对比文件2分析.md
    └── ...
```

### 8.3 用户最终确认

系统 **SHALL**：
1. 展示完整答复包
2. 请用户最终确认
3. 用户可进行修改

---

## 9. Step 8: 自我反思与持续改进

### 9.1 对照用户修改

当用户对答复进行修改后，系统 **SHALL**：
1. 识别用户修改的具体内容
2. 对比原版本和修改版本

### 9.2 分析与反思

系统 **SHALL** 执行：

| 分析项 | 内容 |
|--------|------|
| 修改原因 | 为什么用户要做这个修改？ |
| 遗漏分析 | 为什么系统没有考虑到这点？ |
| 规律总结 | 是否存在可复用的规律？ |

### 9.3 改进建议

系统 **SHALL** 提出改进建议：
1. 更新答复流程
2. 优化模板
3. 补充知识库
4. 调整策略选择逻辑

### 9.4 用户确认改进

系统 **SHALL**：
1. 展示改进建议
2. 说明预期效果
3. 请用户确认是否采纳
4. 根据确认结果更新系统

| 指标 | 标准 |
|------|------|

| 技术描述准确率 | ≥ 90% |
| 用户修改率 | ≤ 20% |

## 12. 使用指南

### 12.1 快速开始

```bash
# 1. 进入项目目录
cd ~/Athena工作平台/openspec-oa-workflow

# 2. 创建新的答复任务
openspec new change oa-response-YYYYMMDD-案件号

# 3. 上传材料，启动流程
# 在 Claude Code 或 OpenCode 中：
# "请根据审查意见通知书开始答复流程"

# 4. 按流程逐步确认

# 5. 归档
openspec archive oa-response-YYYYMMDD-案件号
```

### 12.2 命名规范

```
oa-response-YYYYMMDD-申请号
```

示例：
```
oa-response-20260326-CN202410001234
```
